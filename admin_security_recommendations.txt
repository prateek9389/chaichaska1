SECURITY UPGRADE PLAN FOR ADMIN & CHAIMAKER PANELS
======================================================
While we have locked down the UI to prevent unauthorized access, a highly technical user could potentially bypass the frontend and talk directly to your database. 

To achieve Enterprise-level security, you should implement the following 5 steps:

1. SECURE FIREBASE FIRESTORE RULES (CRITICAL)
------------------------------------------------------
Right now, your security is enforced entirely on the "client-side" (in the browser). If your Firestore rules are open (e.g., `allow read, write: if request.auth != null;`), ANY logged-in customer could use custom code to delete products or view orders.
* FIX: Update your Firebase Firestore rules in the Firebase Console so that only your specific admin email can write to sensitive collections.
* Example Rule:
  match /products/{document} {
    allow read: if true;
    allow write: if request.auth != null && request.auth.token.email == "rohitsengar02@gmail.com";
  }

2. USE FIREBASE CUSTOM CLAIMS (BEST PRACTICE)
------------------------------------------------------
Instead of checking `user.email == "rohitsengar02@gmail.com"` everywhere in your frontend code, you can use a Firebase Cloud Function to assign an "Admin" badge (called a Custom Claim) securely to your account.
* Why? It makes your Firestore rules cleaner and prevents spoofing.

3. NEXT.JS MIDDLEWARE ROUTE PROTECTION
------------------------------------------------------
Currently, the browser downloads the entire Admin panel code, and THEN the `useEffect` realizes the user shouldn't be there and kicks them out. 
* FIX: Create a `middleware.js` file in your Next.js root. This intercepts the user on the server. If they aren't the admin, the server won't even send them the Admin page code—it will redirect them instantly.

4. REMOVE HARDCODED PASSWORDS
------------------------------------------------------
Your code still contains references to `admin@chaichaska.com` and `process.env.NEXT_PUBLIC_ADMIN_PASSWORD`. 
* FIX: Since you are now exclusively using Google Sign-In for the admin, you should completely delete the old email/password input forms from the Admin and Chaimaker login screens. This reduces the "attack surface".

5. SEPARATE BREWMASTER & ADMIN ROLES
------------------------------------------------------
Currently, `rohitsengar02@gmail.com` is used for both Admin and Chaimaker. 
* FIX: In the future, if you hire staff to run the kitchen, you don't want them having Admin access (where they can see revenue or delete users). You should create a second specific Google account (e.g., `kitchen@chaichaska.com`) and restrict the Chaimaker panel to ONLY that email, while keeping the Admin panel restricted to yours.
